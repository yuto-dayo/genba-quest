v3 の決定事項

  • FAB は既存実装を流用 ── ラベル付きメニュー (経費 / 売上 / 請求書) がモバイルで右下に。ドラッグ→端スタッシュも既存挙動のまま
  • 取引先設定は Settings 配下 ── 低頻度操作、他のマスタ管理と並べる
  • 取引先カードに次の締め日 + 入金予定日 ── 「あと何日」が一目に。weekly 対応でルール表示も pill 化
  • 経費入力で「現場 / 本社」必須 + 立替トグル ── billable トグルは現場選択時のみ、デフォルトON、材工一式は自動OFF (将来は site のフラグから自動判定)
  • まとめ請求ドラフトは売上 + 諸経費の2セクション ── 立替経費を一覧で含む/外す。「材工一式 (自動OFF)」のように理由も表示
  • 取引先設定にルール変更履歴 ── 過去の取り決めが残る = 番頭レス可視性

既存ロジックの流用マッピング

  • ヒーロー手残り → fetchPL().profit そのまま
  • 取引リスト → fetchTransactions() + vendor_name 既存フィールド
  • 親方チェック待ち → fetchPendingApprovals()
  • 取引先カード合計 → 新規 API /accounting/partners/summary?month=YYYY-MM (server で client_id GROUP BY)
  • 次の締め日 + 入金予定 → nextBillingPeriod(client) (新規ヘルパー)
  • まとめ請求ドラフト → 新規 API /accounting/invoice-drafts/build?client_id=X&period=YYYY-MM

削った / 諦めたもの (80点ライン)

  • カレンダーへの締め日マーカー連携 → Phase 2
  • 稼働日ベース按分・shared_expense_pools → そもそも按分やめた
  • ルール変更時の「過去 period への影響」シミュレーター → 過去不変原則で問題出にくい
  • OCR 自動撮影 (QuickBooks Receipt Snap 風) → 既存OCR を活かす
  • Sherpa からの請求漏れ通知 → 後回し、まずカウンタが動けば足りる
