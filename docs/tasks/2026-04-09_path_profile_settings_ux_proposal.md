# 2026-04-09 LUQO / PATH UIUX 改善提案

## 結論

- 自分のスキル・Level・認定状態は `/settings` に寄せる
- PATH 画面は「今月の評価作業」に集中させる
- 詳細技能は常時表示しない。必要時だけ検索できればよい

この分離で、職人にとっての主な疑問を整理できる。

- 「自分はいま何ができるのか」→ `/settings`
- 「今月なにを入力・確認すればよいか」→ PATHレビュー画面

## 背景

現状の PATH 画面は、以下を同じ場所で扱っている。

- 月末フォーム入力
- AI整理
- 評価確定
- 報酬計算
- 技能認定
- LUQO 比較
- current profile の参照

制度としては整合しているが、一般的な職人目線では認知負荷が高い。

- 何から触ればよいか分かりにくい
- 自分の恒常情報と今月の作業情報が混ざっている
- 専門用語が多く、運用者の頭の中に前提知識が必要

## PM向け判断

### 1. `/settings` を「本人プロフィール」の置き場所にする

妥当。既存の `/settings` は現在「請求書設定」「取引先マスタ」の集約画面だが、情報の性質としては「設定・マスタ管理」の置き場になっている。

ここに本人プロフィールを追加すると、職人は「自分の現在地」を毎回同じ場所で確認できる。

### 2. PATH画面からプロフィール常設表示を外す

PATH画面に必要なのは「今月の評価に必要な最小情報」だけでよい。

残すもの:

- 今月の入力状況
- AI整理候補
- 今月の評価確定
- 今月の報酬確認

移すもの:

- current Level
- 6つの主評価項目の現在値
- 認定済み技能
- 認定履歴
- 最近の評価履歴

### 3. 詳細技能は検索前提でよい

常時一覧は不要。職人は普段「全部の技能一覧」を見たいわけではなく、

- 自分が認定済みか
- 次に何を目指すか
- 特定の技能があるか

だけ分かれば十分なことが多い。

そのため、プロフィール画面では以下で足りる。

- 認定済み技能の要約
- 最近追加された技能
- 検索入口

## 情報設計

### `/settings`

`/settings` は以下の 3 セクション構成を推奨する。

1. マイプロフィール
2. 請求書設定
3. 取引先マスタ

優先度は「マイプロフィール」を先頭に置く。
理由は、職人が最も高頻度で見たい個人情報だから。

### マイプロフィールで見せる内容

#### 上部サマリー

- 氏名
- 現在 Level
- 今月の評価状態
  - 例: 未提出 / 入力済み / レビュー待ち / 確定済み
- 認定技能数

#### 任せられること

- 6つの主評価項目の現在値
- 一言での説明
  - 例: 「品質安定は安定自走」
  - 例: 「段取りは条件付きで単独可」

#### 認定技能

- 認定済み技能のカード一覧
- カテゴリ絞り込み
- 技能名検索
- 「最近認定されたもの」表示

#### 最近の評価履歴

- 月
- Level
- 主な変化
- コメント要約

## PATH画面の役割再定義

PATH画面は「プロフィール閲覧画面」ではなく「今月の月次評価ワークフロー画面」にする。

### 画面の job

- 評価担当者が、今月分のレビューを迷わず完了する

### 画面の P0

- 今月の評価確定を終える

### 画面の推奨構成

1. 対象メンバー一覧
2. 今月の状態サマリー
3. 月末フォーム
4. AI整理確認
5. 評価確定

7. 詳細機能

「技能認定」「LUQO比較」は初期表示から外し、詳細セクションへ逃がす。

## `/settings` ワイヤー案（2026-04-09）

### 結論

`/settings` の先頭に「マイプロフィール」を置き、そこを**自分の現在地と技能確認の入口**にする。

- `/settings` は「見る・確認する」場所
- PATH は「今月の評価を進める」場所
- 技能認定の起票や評価確定は PATH に残す

### 情報階層

1. マイプロフィール
2. 今月の評価状況
3. 認定技能
4. 技能を探す
5. 最近の評価履歴
6. 組織の設定（請求書設定 / 取引先マスタ）

### first view に置く内容

375px 幅では、最初の 1 画面で以下が分かる状態を目標にする。

- 誰の画面か
- 今どの Level か
- 今月の評価がどこまで進んでいるか
- 6つの主評価項目が今どの状態か
- 認定技能の全体感があるか
- 詳しい技能確認へ進めるか

### セクション別の役割

#### 1. マイプロフィール

- ページ先頭に固定
- `氏名 + 現在Level + 今月の評価状態` をまとめて見せる
- ここで「自分の今の位置」が 3 秒以内に理解できることを優先する

#### 2. 6つの主評価項目

- カード一覧ではなく、**1行ラベル + 現在値** の縦並びにする
- first view では詳細説明を見せすぎない
- 状態表現は内部用語でなく職人向けの短い文言を使う

#### 3. 認定技能

- 一覧を最初から全部見せない
- `認定済み件数` と `代表的な技能 2-3 個` だけを first view に出す
- 詳細確認は「技能を探す」から入る

#### 4. 技能を探す

- first view では検索入口だけ置く
- 遷移先または展開先で検索 UI を出す
- 認定済み / 候補 / 未認定 を絞れるとよい

#### 5. 最近の評価履歴

- first view では最新 2-3 件に絞る
- 月、Level 変化、再確認の有無が一目で分かればよい
- 根拠詳細や確認ログは second view に回す

#### 6. 組織の設定

- 既存の請求書設定と取引先マスタはこの下に残す
- 見出しを「組織の設定」に変え、個人プロフィールと明確に分ける
- first view の主役にしない

### PATH から移すもの / 残すもの

#### `/settings` に寄せるもの

- `Current Profile`
- 認定済み技能の常設表示
- 最近の技能認定の参照導線

#### PATH に残すもの

- 月末フォーム
- AI整理
- 評価確定
- 報酬確認
- 技能認定 Proposal の起票

### 文言方針

- `Current Profile` → `マイプロフィール`
- `最近の技能認定` → `最近の認定`
- `skill proposal を起票` は `/settings` では見せず、PATH 側だけに残す
- 英語ラベルは避け、職人が一読で意味を取れる日本語に揃える

### 実装時の判断メモ

- first view の情報量が多すぎる場合は、「認定技能」と「最近の評価履歴」を折りたたみで下に逃がす
- ただし `氏名 / Level / 今月の評価状態 / 6つの主評価項目` は常時表示を維持する
- PC では 2 カラム化してよいが、モバイル基準で構造を決める
