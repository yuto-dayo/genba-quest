# 不変核 vs 可変形 — 分類リファレンス

## 絶対不変核（3つのみ）

これを変えるとGENBA QUESTのDAO性が壊れる。コードにゲートとして実装必須。

| 不変核 | コード上の実装場所 | 壊れるとどうなるか |
|--------|------------------|------------------|
| AI自己承認禁止 | `PolicyEngine.ts:272` — `AI_SELF_APPROVAL_PROHIBITED` | AIが自分で承認→組織ガバナンス崩壊 |
| Proposal中心 | 全routeが `proposals` テーブルに書く | 状態変更の追跡不能・監査不可 |
| Ledgerバランス | `ProposalService.ts` — 借方=貸方チェック | 財務データの整合性崩壊 |

## 可変形（フェーズや文脈で変えてよい）

### 金額閾値（最も頻出の誤分類）

```
❌ 不変と思われがちだが → ✅ 可変
5,000円, 30,000円, 50,000円 の閾値はすべてDBのpoliciesテーブルで管理すべき
```

**現状の問題箇所（技術的負債）:**

| ファイル | 行 | 問題 | 修正方針 |
|---------|-----|------|---------|
| `server/src/routes/accounting.ts` | 170-171 | `total > 30000` ハードコード | PolicyEngine経由に変更 |
| `server/src/routes/accounting.ts` | 240,307 | `approval_limit ?? 50000` | DBまたはenv変数から取得 |
| `server/src/services/ocrService.ts` | 305,311 | `totalAmount > 30000/5000` | PolicyEngine経由に変更 |

### org_id（環境変数で管理すべき）

```
❌ 不変と思われがちだが → ✅ 可変（環境ごとに変わる）
'00000000-0000-0000-0000-000000000001' のハードコード
```

**問題箇所:**

| ファイル | 行 | 問題 | 修正方針 |
|---------|-----|------|---------|
| `PolicyEngine.ts` | 127 | コンストラクタデフォルト値 | `process.env.DEFAULT_ORG_ID` を使う |
| `ProposalService.ts` | 116 | 同上 | 同上 |
| ルート系 | 各所 | `process.env.DEFAULT_ORG_ID \|\| '000...'` | ✅ 既に正しい形 |

### 承認フロー（Policyで表現すべき）

```
可変: 承認人数・自動承認条件・AIが承認できるか
不変: Proposalを経由すること自体
```

## フェーズ別の「今の最適形」

フェーズが変わると「最適形」も変わる。ディレクターはフェーズを意識して決定する。

| 判断事項 | A-0(今) | A-1 | B | C | D |
|---------|---------|-----|---|---|---|
| 閾値管理 | DBへ移行中 | DB完全管理 | 同左 | 同左 | 動的変更可 |
| UIの精度 | 粗くてよい | 粗くてよい | 改善 | 刷新 | 完成 |
| Sherpa | 基本のみ | 同左 | フル統合 | 同左 | 高度化 |
| テスト | 主要パス | 全フロー | E2E | 回帰 | 網羅 |
